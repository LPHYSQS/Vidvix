# Vidvix AI UI 验证报告

## 文档信息

- 轮次：`R3`
- 日期：`2026-04-23`
- 执行 Agent：`Codex`
- 验证对象：`Views/MainWindow.xaml`、`Views/MainWindow.Chrome.cs`、`Views/AiPage.xaml`、`Views/MergePage.xaml`、`App.xaml`
- 构建基线：`dotnet build .\Vidvix.sln -c Debug -v minimal` 通过，`0` 警告、`0` 错误
- 运行基线：Debug 产物成功启动；窗口 DPI `144`，等效缩放 `150%`；启动后 `8s` 内未黑屏、白屏或闪退

## 验证方法

- 运行态验证：启动 `bin\x64\Debug\net8.0-windows10.0.19041.0\Vidvix.exe`，记录 `1400x900` 与 `1100x900` 两组窗口表现。
- 自动化验证：使用 UIAutomation 校验左侧工作区 toggle 的 `ToggleState`、可用性与当前选中项。
- 结构审查：按 `UTF-8` 阅读 `Views/MainWindow.xaml`、`Views/MainWindow.Chrome.cs`、`Views/AiPage.xaml`、`Views/MergePage.xaml`、`App.xaml`。
- 说明：当前环境实际运行在 `150%` 缩放；`100%` / `125%` 结论基于现有 XAML 布局、列定义和运行结果做等效审查，用于驱动 `R4`，不是系统级 DPI 切换实测。

## 结论摘要

- `AI` 导航入口已正确接入，页面可进入，应用启动稳定。
- `AI` 按钮在自己被选中时，没有进入与既有模块一致的蓝色响应态，结论为：**不一致，必须修**。
- `AI` 图标结论为：**保留图形语义，但需要微调**；当前不建议直接更换。
- `AI` 页面当前只对齐到了“三栏骨架”这一层；页内标题层级、列宽策略、控件密度与合并模块仍明显不一致。
- 在当前机器的 `150%` 缩放与 `1100x900` 窄窗下，AI 页面右侧列已经出现明显裁切风险，结论为：**必须修**。

## 导航专项验证

| 项目 | 结果 | 记录 |
| ---- | ---- | ---- |
| 选中态 | Failed | UIAutomation 显示 `AiWorkspaceToggle = On` 时，运行态截图中 `AI` 图标与文本仍为白色；切到 `Merge` 后，`合并` 图标与文本会进入蓝色选中态。`Views/MainWindow.Chrome.cs` 的 `UpdateWorkspaceToggleVisuals()` 已处理 `Video / Audio / Trim / Merge / SplitAudio / Terminal`，但漏掉了 `AiWorkspaceIcon` 与 `AiWorkspaceText`。 |
| 悬停态 | Review Needed | `App.xaml` 的 `WorkspaceModeToggleButtonStyle` 声明了 `PointerOver`，但没有实际动画或画刷切换。AI 与既有模块表现一致，但反馈偏弱。 |
| 按下态 | Review Needed | `Pressed` 同样只有空状态，没有单独的视觉反馈。该问题是共享模板问题，不是 AI 独有回归。 |
| 禁用态 | Passed | `Disabled` 状态会把 toggle 整体降到 `0.45` 透明度；AI toggle 继续复用与既有模块相同的 `IsEnabled="{x:Bind ViewModel.CanModifyInputs}"` 规则。此轮未强行制造运行中禁用场景，但代码路径一致。 |
| 图标尺寸 / 重心 / 清晰度 | Review Needed | AI 图标使用 `18x18`、`StrokeThickness="1.8"` 的星芒路径；与合并 / 拆音 / 终端的 `1.9` 线宽图标相比，视觉重心略偏上、光学重量略轻。可读性尚可，但完成度未达到同级。结论：**微调，不更换**。 |

### AI 图标结论

- 结论：`微调`
- 保留原因：图形语义明确，能够和“AI”文案形成稳定识别，不必为了首发范围再引入一轮完全替换。
- 微调方向：略增线宽，降低图形重心，减少顶部轻飘感；保持与左侧其他线性图标相近的完成度。

## AI 页面与合并模块对标

| 对标项 | 合并模块现状 | AI 页面现状 | 结论 |
| ---- | ---- | ---- | ---- |
| 整体壳层 | 直接进入三栏工作区主体 | 在三栏主体前又加了一张页内 hero 卡 | 不一致；AI 页内标题层级重复 |
| 外层列宽策略 | `260 / * / 320`，侧栏宽度稳定 | `* / 1.15* / *`，三列会一起被压缩 | 不一致；不利于后续接入真实控件 |
| 左列定位 | 素材库标题、导入按钮、滚动列表 | 只有标题与说明文字 | 部分一致；结构在，但密度明显偏空 |
| 中列定位 | 核心工作区与多层卡片交互 | 说明文字 + 两张占位卡 | 部分一致；结构存在，但与合并模块的工作区重心不一致 |
| 右列定位 | 输出设置控件栈 + 滚动容器 | 只有标题与说明文字 | 部分一致；结构在，但后续控件承载策略尚未对齐 |
| 标题层级 | 主窗口壳层标题承担模块标题 | 主窗口壳层标题之外，页内再次出现 `AI 工作区 / AI 模块骨架已接入` | 不一致；信息重复，垂直空间被提前吃掉 |
| 卡片层级 | 外卡 + 内嵌 inset 卡有明确主次 | 三栏外卡存在，但内容仍以说明文案为主 | 部分一致；视觉语言相近，但完成度明显偏空壳 |

### 已对齐的点

- 已使用共享的 `SurfaceCardBorderStyle`、`SectionTitleTextStyle`、`CaptionTextStyle`。
- 已沿用左 / 中 / 右三栏的大方向，没有把 AI 页面做成与既有模块完全割裂的新壳层。
- 中列内的 `AI补帧` / `AI增强` 占位卡保留了后续模式区的落点，适合 `R5+` 增量接线。

### 未对齐的点

- 主窗口壳层已经有模块标题，AI 页面再次放一张完整 hero，形成重复标题层级。
- AI 页面列宽没有对齐合并模块的固定侧栏策略，导致未来接入列表和输出控件时容易重排。
- AI 页面当前更像“说明页”，不是“待接功能的工作区壳层”；与合并模块的工作区感仍有差距。

## 问题清单

### 必须修

1. `AI` 导航选中态缺少与既有模块一致的蓝色反馈。
   这是 `R3` 明确要求确认的验收项，且已经在运行态与代码路径中双重确认。

2. AI 页面存在与主窗口壳层重复的页内 hero。
   当前主窗口已经展示 `当前模块 / AI 工作区 / 描述`，AI 页内再次放同等级标题卡，会稀释工作区主体并挤压垂直空间。

3. AI 页面三列布局在 `150% + 1100x900` 下缺少有效自适应。
   当前右侧列已有裁切风险；如果继续在这个壳层上叠加素材列表、模式切换和输出控件，返工成本会更高。

4. AI 页面外层列宽策略未对齐合并模块。
   `R5` 之前应先收口为可承载真实控件的稳定壳层，否则后续功能落位会反复改布局。

### 建议修

1. AI 图标保留，但需做一次线宽 / 重心微调。
2. 如果 `R4` 会动到共享导航模板，可顺手为 hover / pressed 增加轻量反馈；若改动面扩大，可延后到统一导航整理轮次。

### 可延后

1. 顶部命令栏目前只有“应用设置”，视觉上偏空，但这是功能冻结下的预期，不应在 `R4` 偷带 `R5` 功能。
2. 素材列表、模式切换、输出控件的真实交互仍属于 `R5`，本轮只确认壳层是否适合继续承载。

## 对 R4 的直接建议

- 只修本报告已确认的 UI 问题，不追加素材导入、模式状态、输出设置或 runtime/workflow。
- 优先顺序建议如下：
  1. 修 `AI` 导航选中态与图标选中逻辑。
  2. 收口页内标题层级，去掉或降级重复 hero。
  3. 把 AI 页面三栏布局拉回到与合并模块同级的壳层策略。
  4. 回填 `100% / 125% / 150%` 与窄窗验证结论。

## 阻断与非阻断划分

- 阻断继续功能开发：
  - `AI` 导航选中态不一致
  - 页内重复 hero
  - `150%` 窄窗下三列压缩与右列裁切风险
  - 外层列宽策略未对齐合并模块
- 非阻断，可后续跟进：
  - AI 图标微调
  - 共享导航 hover / pressed 反馈增强
  - 顶部命令栏视觉饱满度
