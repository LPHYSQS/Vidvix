# 裁剪模块新增音频裁剪功能任务目录

## 文档目的

这份文档不是功能说明书，而是给下一任继承者的实施路线图。

目标非常明确：

1. 保持现有“视频裁剪”能力全部可用，不回退、不改味。
2. 在同一个裁剪模块里，新增“音频裁剪”这条并行能力。
3. 不新增导入按钮，继续使用当前唯一的“导入文件”入口。
4. 继续坚持“单次只导入 1 个文件”的裁剪模型。
5. 优先降低耦合，避免再把“视频专用假设”继续硬塞进未来的音频链路。

这次扩展不应该被理解成“在现有视频裁剪代码里多塞几个 if”，而应该理解成：

- 复用现有裁剪模块的通用壳层
- 拆出“媒体类型差异”
- 让视频裁剪和音频裁剪共用一套交互骨架，但各自有独立的导入判定、预览呈现、输出格式和导出命令策略

## 当前裁剪模块的真实结构

### 1. 组合根

当前裁剪模块的依赖装配在 `Utils/AppCompositionRoot.cs`。

- `VideoTrimWorkspaceViewModel`
- `VideoTrimWorkflowService`
- `MpvVideoPreviewService`
- `MediaInfoService`
- `FFmpegService`
- `VideoTrimCommandFactory`

也就是说，当前裁剪模块是“一条视频专线”：

- View 只认 `VideoTrimWorkspaceViewModel`
- ViewModel 只认 `IVideoTrimWorkflowService` 和 `IVideoPreviewService`
- WorkflowService 只认“视频导入 + 视频导出”
- CommandFactory 只认“视频裁剪输出格式”

### 2. 当前关键文件地图

建议后续实现时先围绕这些文件工作：

- `Views/Controls/VideoTrimWorkspaceView.xaml`
  - 裁剪模块主界面
  - 标题、预览区、播放控制、时间轴、裁剪区间、输出设置、信息区都在这里
- `Views/Controls/VideoTrimWorkspaceView.xaml.cs`
  - 时间轴点击/拖拽、预览宿主窗口布局、播放控制按钮事件都在这里
- `ViewModels/VideoTrimWorkspaceViewModel.cs`
  - 导入、状态、文案、信息字段、输出格式、输出路径、导出、偏好保存
- `ViewModels/VideoTrimWorkspaceViewModel.Preview.cs`
  - MPV 预览控制、时间轴优先级、拖拽/点击、边界预热、播放停止逻辑
- `Services/VideoTrimWorkflowService.cs`
  - 当前导入校验只允许视频，导出流程也只面向视频
- `Services/FFmpeg/VideoTrimCommandFactory.cs`
  - 当前裁剪导出命令工厂，只支持视频裁剪输出
- `Services/FFmpeg/MediaProcessingCommandFactory.cs`
  - 这里已经存在成熟的音频输出格式编码映射，后续应复用，而不是重复抄一份
- `Core/Models/ApplicationConfiguration.cs`
  - 当前视频裁剪输入格式、视频裁剪输出格式、音频输入格式、音频输出格式的源头都在这里
- `Utils/MediaPathResolver.cs`
  - 裁剪输出路径规划在这里，当前其实已经足够通用
- `Core/Models/MediaDetailsSnapshot.cs`
  - 当前媒体探测结果模型，本身就支持 `HasVideoStream` / `HasAudioStream`

## 当前裁剪模块的真实流程

### 一条完整的视频裁剪链路

1. 用户进入裁剪模块，看到 `VideoTrimWorkspaceView.xaml` 的占位态界面。
2. 点击“导入文件”或拖拽文件。
3. `MainWindow.xaml.cs` 的拖拽入口会在裁剪模块下转发到 `TrimWorkspace.ImportPathsAsync(paths)`。
4. `VideoTrimWorkspaceViewModel.ImportPathsAsync(...)` 做第一层状态校验：
   - 正在导出时不允许导入
   - 当前已有输入时不允许再导入
   - 之后调用 `VideoTrimWorkflowService.ImportAsync(...)`
5. `VideoTrimWorkflowService.ImportAsync(...)` 做第二层业务校验：
   - 只能 1 个文件
   - 不支持文件夹
   - 扩展名必须在 `SupportedTrimInputFileTypes`
   - `MediaInfoService` 探测后，必须有视频流
   - 时长必须有效
6. 导入成功后，ViewModel 写入：
   - `InputPath`
   - `InputFileName`
   - `MediaDetailsSnapshot`
   - `MediaDuration`
   - `SelectionStart = 0`
   - `SelectionEnd = duration`
   - `CurrentPosition = 0`
7. `VideoTrimWorkspaceView.xaml.cs` 监听 `InputPath`，触发 `ReloadPreviewAsync()`。
8. `VideoTrimWorkspaceViewModel.Preview.cs` 调用 `IVideoPreviewService.LoadAsync(...)`，由 MPV 加载预览。
9. MPV 就绪后，通过 `MediaOpened` 事件通知 ViewModel：
   - 设置预览可用
   - 同步时长
   - 开启边界预热
10. 用户开始：
   - 播放 / 暂停
   - 点击时间轴跳转
   - 拖拽时间轴预览
   - 修改入点/出点
   - 跳转到入点 / 出点
11. 导出时：
   - ViewModel 生成 `VideoTrimExportRequest`
   - WorkflowService 确保 FFmpeg 运行时存在
   - WorkflowService 根据偏好决定是否探测 GPU 视频编码
   - `VideoTrimCommandFactory` 生成 FFmpeg 命令
   - `FFmpegService.ExecuteAsync(...)` 执行

## 当前已经具备、可以直接复用的资产

这些能力后续做音频裁剪时不应该重写：

### 1. 文件选择器本身已经足够通用

`Services/FilePickerService.cs` 只接受 `FilePickerRequest.AllowedFileTypes`。

这意味着：

- 不需要改文件选择器服务本体
- 只需要给裁剪模块一个“视频扩展名 + 音频扩展名”的联合列表

### 2. 媒体探测模型已经支持音频

`MediaDetailsSnapshot` 已经有：

- `HasVideoStream`
- `HasAudioStream`
- `OverviewFields`
- `VideoFields`
- `AudioFields`

而且音频转换模块已经依赖这套模型，所以媒体探测层并不是问题，问题只是裁剪模块目前把它当成“必须有视频流”来用。

### 3. 输出路径工具已经够通用

`Utils/MediaPathResolver.CreateTrimOutputPath(...)` 只基于输入文件路径、输出扩展名和输出目录工作。

它并不关心输入是视频还是音频。

这意味着：

- 音频裁剪不一定需要新建一个 `CreateAudioTrimOutputPath`
- 继续使用 `_trim` 后缀完全可行
- 真正要改的不是路径工具，而是界面文案里“原视频所在文件夹”的字样

### 4. 音频格式源头已经存在

`Core/Models/ApplicationConfiguration.cs` 里已经有：

- `SupportedAudioInputFileTypes`
  - 当前代码真实值为：`.mp3`, `.m4a`, `.aac`, `.wav`, `.flac`, `.wma`, `.ogg`, `.opus`, `.aiff`, `.aif`, `.mka`
- `SupportedAudioOutputFormats`
  - 当前音频转换模块的输出格式源头

注意：

- 这份代码里的真实扩展名是 `m4a / wav / aiff`
- 如果产品后续确认还要额外支持 `mca`、`waa` 之类当前代码中不存在的扩展名，必须只在配置层补，不要散落到 ViewModel、XAML 文案或命令工厂里临时硬编码

### 5. 裁剪交互骨架本身是通用的

当前这些交互对音频裁剪依旧成立：

- 播放 / 暂停
- 时间轴点击跳转
- 时间轴拖拽预览
- 入点 / 出点
- 跳转到入点 / 出点
- 音量调整
- 输出目录
- 计划输出路径
- 导出进度

也就是说，应该复用的是“交互骨架”，而不是复制一份新的音频裁剪页面。

## 当前阻碍音频裁剪的核心耦合点

下面这些地方是后续一定要拆的“视频专用假设”。

### 1. 类型命名全部是视频专用

当前关键类型都叫：

- `VideoTrimWorkspaceViewModel`
- `VideoTrimWorkflowService`
- `VideoTrimExportRequest`
- `IVideoTrimWorkflowService`
- `IVideoPreviewService`
- `VideoTrimCommandFactory`

这不是马上必须全量重命名，但它说明当前代码的语义边界就是“视频裁剪”。

### 2. 导入校验是视频专用

`VideoTrimWorkflowService.ImportAsync(...)` 当前强制要求：

- 扩展名在 `SupportedTrimInputFileTypes`
- `snapshot.HasVideoStream == true`

所以音频文件在这里会被直接拦掉。

### 3. 输出格式是视频专用

`VideoTrimWorkspaceViewModel` 构造时直接把：

- `AvailableOutputFormats = _configuration.SupportedTrimOutputFormats`

这意味着当前裁剪模块从初始化那一刻起就默认自己只会处理视频输出格式。

### 4. 导出命令工厂是视频专用

`VideoTrimCommandFactory` 当前的命令拼装逻辑明显面向视频：

- `-map 0:v:0?`
- 视频编码策略
- 视频容器输出格式
- GPU 视频编码探测

它不适合直接通过几个 `if` 变成音频裁剪总工厂。

### 5. 预览呈现是视频专用

`VideoTrimWorkspaceView.xaml` 的上方预览区当前默认是视频预览区。

而用户对音频裁剪的要求是：

- 保留底部的播放、跳转、时间轴、音量控制
- 但上方预览区域改成波形图 / 波浪图

这意味着“播放传输”和“预览呈现”必须拆开看。

### 6. 文案目前大量硬编码为视频

当前有不少写死的“视频”文案，例如：

- `Views/Controls/VideoTrimWorkspaceView.xaml`
  - `视频裁剪`
  - `视频预览`
  - `截取视频片段`
  - `原视频所在文件夹`
  - `视频信息`
- `Views/MainWindow.xaml`
  - 顶部仍有 `移除视频`
- `ViewModels/VideoTrimWorkspaceViewModel.cs`
  - `请先导入一个视频文件`
  - `已移除当前视频`
  - `当前已有视频`
  - `正在解析视频信息`
  - `导入视频`
- `ViewModels/VideoTrimWorkspaceViewModel.Preview.cs`
  - `正在准备视频预览`
  - `当前视频无法预览，但仍可尝试直接导出`
- `Services/VideoTrimWorkflowService.cs`
  - `裁剪模块一次只能导入 1 个视频文件`
  - `当前文件不包含可裁剪的视频流`

这些文案不能继续散在各处，否则音频裁剪会变成“功能能用，文案全错”。

## 推荐的低耦合改造原则

### 原则 1：不要一上来就做全仓重命名

第一轮实现时，不建议先把所有 `VideoTrim*` 全部重命名成 `Trim*`。

原因：

- diff 太大
- 回归范围太广
- 很容易把现在已经修好的时间轴/预览优先级问题重新搅乱

更稳的做法是：

1. 先引入新的抽象层
2. 让旧类通过新抽象承接音频能力
3. 功能稳定以后，再决定是否做命名清理

### 原则 2：先把“媒体类型”升格成一等概念

建议新增一个明确的类型，例如：

`TrimMediaKind`

- `Video`
- `Audio`

后续所有差异都围绕它展开，而不是在代码里到处写：

- `if (snapshot.HasVideoStream) ...`
- `if (extension == ".mp3") ...`
- `if (title contains 视频) ...`

### 原则 3：播放传输和预览呈现必须拆开

音频裁剪不是“没有画面的简化视频裁剪”。

后续建议分成两层：

1. 播放传输层
   - 负责加载、播放、暂停、跳转、当前位置、音量
2. 预览呈现层
   - 视频时显示 MPV 视频画面
   - 音频时显示波形图

这样做以后：

- 时间轴、播放、跳转逻辑可以共用
- 预览画面可以按媒体类型切换

### 原则 4：文案不要再硬编码

这次音频裁剪扩展里，文案建议抽成“裁剪工作区展示配置”。

可以是一个新的 profile，也可以是 `ViewModel` 的一组动态只读属性，但目标一定要一致：

- 标题
- 描述
- 占位文案
- 导出说明
- 输出目录提示
- 信息区标题
- 错误提示
- 文件选择提交按钮文字

统一从“当前媒体类型”推导。

## 建议的架构落点

下面这套落点是兼顾“低耦合”和“低风险”的建议。

### 1. 新增一个媒体类型枚举

建议新增：

- `Core/Models/TrimMediaKind.cs`

用途：

- 当前导入的是视频还是音频
- 当前标题显示什么
- 当前输出格式列表用哪套
- 当前预览区显示视频还是波形
- 当前导出命令走哪条策略

### 2. 把导入结果变成“识别媒体类型”的结果

当前 `VideoTrimImportResult` 只有：

- 路径
- 文件名
- Snapshot
- Duration

后续建议让导入结果明确告诉 ViewModel：

- `MediaKind = Video / Audio`

可选路线有两种：

1. 新建 `TrimImportResult`
2. 先在现有 `VideoTrimImportResult` 上补 `MediaKind`

从风险角度看，第二种改动更小。

### 3. 将裁剪工作流从“视频专线”改成“媒体裁剪总入口 + 按类型分发”

推荐方向：

- 新增 `ITrimWorkflowService` 或 `IMediaTrimWorkflowService`
- 内部根据 `TrimMediaKind` 分发到：
  - 视频导入/导出策略
  - 音频导入/导出策略

不推荐做法：

- 在 `VideoTrimWorkflowService` 里一路塞 `if (audio)`
- 然后把所有逻辑都堆在一个类里

### 4. 导出命令建议拆成策略，而不是硬塞进 `VideoTrimCommandFactory`

推荐：

- `IVideoTrimCommandFactory`
- `IAudioTrimCommandFactory`

或者一个统一接口，内部按 `TrimMediaKind` 分派。

不推荐：

- 继续在 `VideoTrimCommandFactory` 里加大量音频分支

原因很简单：

- 视频裁剪关注视频编码、封装、GPU
- 音频裁剪关注音频编码、容器、是否 copy、是否保留元数据

这两类规则长期看不是一回事。

### 5. 音频输出编码规则不要重复写

后续音频裁剪输出格式，建议直接复用 `MediaProcessingCommandFactory` 里现有的音频编码映射经验。

最佳做法：

- 提取一个共享的“音频输出命令策略”帮助类
- 音频转换模块和音频裁剪模块都走它

这样可以避免：

- `.mp3` 在两个地方码率不一致
- `.mka` 在两个地方一个 copy 一个转码
- 后续新增格式时要改两处

### 6. 预览层推荐新增“波形服务”，不要拿 MPV 直接硬装波形

推荐新增一个独立能力，例如：

- `IAudioWaveformService`

它的职责应该是：

- 根据音频文件生成波形数据或波形图缓存
- 支持取消
- 支持按输入文件维度缓存
- 不参与播放控制
- 不写入 `CurrentPosition`

这是关键：

波形服务只负责“画什么”，不能负责“播到哪”。

### 7. MPV 更适合继续当音频播放传输层，不适合承担波形 UI

当前 MPV 预览服务已经很擅长做这些事：

- Load
- Play
- Pause
- Seek
- Set time-pos
- Volume
- Position changed

所以对于音频裁剪，推荐思路是：

- 继续把 MPV 当成“播放后端”
- 再额外加一个“波形显示前端”

而不是把波形绘制强行塞给 MPV。

## 推荐的分步实施顺序

下面的顺序是为了尽量降低回归风险。

### 第 1 步：先把配置和类型边界立住

建议做这些最小改造：

- 新增 `TrimMediaKind`
- 为裁剪模块准备视频/音频两套显示配置
- 为裁剪模块准备视频/音频两套输出格式来源
- 为裁剪模块准备统一的联合输入扩展名列表

这里有一个很重要的建议：

- `SupportedTrimInputFileTypes` 可以升级成“视频 + 音频联合列表”
- 但内部识别媒体类型时，最终应以 `MediaInfoService` 探测结果为准，不要只依赖扩展名

### 第 2 步：只改导入链路，不先改导出链路

先让裁剪模块做到：

- 仍然只能导入 1 个文件
- 可以导入视频
- 也可以导入音频
- 导入后能识别当前是视频裁剪还是音频裁剪
- 能正确显示标题、文件名、信息卡片、输出格式列表

在这一步，视频行为必须保持原样。

### 第 3 步：把界面文案和信息区做成动态

建议优先改这些点：

- 顶部标题：`视频裁剪` / `音频裁剪`
- 顶部说明：不要再写死“视频预览”
- 占位说明：`导入后会在同一张主卡片中显示文件预览...`
- 右侧选区说明：`导出时会严格按照入点和出点截取所选片段`
- 输出目录提示：
  - 如果产品允许，最稳妥的低耦合文案其实是 `留空时默认导出到原文件所在文件夹`
  - 这比“原视频所在文件夹 / 原音频所在文件夹”双分支更省心
- 信息区标题：
  - 视频时：`视频信息`
  - 音频时：`音频信息`
  - 或者统一成 `文件信息`，但如果要满足产品表达，建议还是按媒体类型动态显示

### 第 4 步：把音频预览改成波形呈现

这是这次扩展里唯一真正新的预览能力。

建议采用两阶段思路：

#### 阶段 A：先做可用版本

- 后台生成一张静态波形图或一组波形采样数据
- 在预览区绘制：
  - 波形
  - 当前播放游标
  - 入点 / 出点遮罩

这样已经能满足大部分“看见音频形状”的需求。

#### 阶段 B：再做更细的性能优化

如果后续还要支持：

- 超长音频
- 更高分辨率波形缩放
- 更精细的局部拖拽反馈

再继续做按宽度分层缓存、抽样等级缓存等优化。

不要一上来就做过度设计。

### 第 5 步：单独补音频裁剪导出策略

音频裁剪导出时应满足：

- 仅导出音频流
- 不映射视频流、字幕流、数据流
- 输出格式走音频格式列表
- 默认输出目录逻辑沿用当前裁剪模块
- 计划输出路径逻辑沿用当前裁剪模块

注意：

- 当前视频裁剪 workflow 里有 GPU 视频编码探测
- 音频裁剪路径不应触发这部分逻辑

### 第 6 步：最后再处理偏好持久化

当前用户偏好里只有：

- `PreferredTrimOutputFormatExtension`
- `PreferredTrimOutputDirectory`
- `PreferredTrimPreviewVolumePercent`

如果后续音频裁剪和视频裁剪要记住不同输出格式，建议新增：

- `PreferredVideoTrimOutputFormatExtension`
- `PreferredAudioTrimOutputFormatExtension`

输出目录可以继续共用，也可以分开，取决于产品需求。

从“减少复杂度”角度看，输出目录继续共用是可以接受的。

## 这次扩展最容易踩的坑

下面这些坑都值得明确提醒继任者。

### 1. 不要让“波形加载”抢占“用户跳转”

当前视频裁剪已经修过一轮“用户点击时间轴优先级低于后台播放位置回写”的问题。

音频裁剪继续扩展时，必须守住这个原则：

- 用户点击 / 拖拽 / 跳入点 / 跳出点的优先级最高
- 后台波形加载不能回写当前位置
- 后台边界预热不能回写当前位置
- 后台旧任务完成时不能覆盖新文件状态

### 2. 不要把“波形已生成”和“可播放”绑成一个状态

推荐至少拆开两个状态：

- 播放后端是否 ready
- 波形前端是否 ready

否则会出现两类坏结果：

- MPV 已经可播，但 UI 因为波形未完成而整体不可用
- 波形已经显示，但播放后端尚未 ready，用户误以为可以立即交互

### 3. 所有异步任务都要带当前会话标识

当前裁剪模块已经有：

- `_sourceVersion`
- MPV load session
- selection boundary warmup cancellation

音频裁剪新增波形后，也要有同样的会话隔离。

否则会出现：

- 移除文件后，旧波形任务又回来把新界面刷掉
- A 文件导入后立刻移除，再导入 B 文件，结果 A 的波形覆盖 B

### 4. 不要只靠扩展名判定媒体类型

正确顺序应该是：

1. 扩展名做快速白名单过滤
2. `MediaInfoService` 做真实媒体探测
3. 以 `HasVideoStream` / `HasAudioStream` 做最终判定

否则会被这些情况打脸：

- 扩展名伪装
- 容器里没有目标流
- 特殊音频容器

### 5. 不要把音频逻辑继续堆进 `VideoTrimCommandFactory`

短期看省事，长期看一定会出问题。

因为一旦后续还要支持：

- 保留封面
- 多音轨选择
- 音频 copy / transcode 策略
- 元数据保留

视频裁剪工厂会迅速失控。

### 6. 不要在第一版就大规模重命名 View / ViewModel / Service 文件

真正影响功能的是“抽象是否正确”，不是文件名是否完美。

先让架构边界正确，再考虑统一命名。

## 对“最小可行版本”的建议

如果下一任继承者希望先稳稳做出第一版，而不是一次把所有细节做到顶格，建议 MVP 定义如下：

1. 同一个导入文件按钮支持视频和音频。
2. 视频路径完全不回归。
3. 音频导入后能识别为“音频裁剪”。
4. 顶部标题、说明、信息卡、输出格式能切换到音频语义。

## 最后的建议

这次功能扩展最重要的不是“把音频导入进来”，而是借这个机会把裁剪模块从“视频专用实现”升级成“媒体裁剪框架”。

但请注意节奏：

- 不要为了追求命名完美，先做大重构
- 不要为了追求波形炫酷，先把时间轴稳定性搞丢
- 不要为了省事，把音频逻辑塞进现有视频导出工厂

最稳的路线永远是：

1. 先抽象媒体类型
2. 再做导入识别
3. 再做动态文案和动态输出格式
4. 再做音频波形
5. 最后补音频导出策略和偏好持久化

只要按这个顺序推进，现有视频裁剪成果就能保住，新增音频裁剪也会比较顺滑。
